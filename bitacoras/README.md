## ¿Qué es una bitácora?

Un documento breve (½ a 1 página) donde explicas las decisiones de diseño que tomaste al refactorizar el legacy. No es un resumen del patrón — es un registro de tu razonamiento.

## ¿Cuándo se entrega?

Una bitácora acompaña cada mini-entrega. Se entrega en el mismo Pull Request que tu código.

## Formato

Usa el archivo `_template.md` de esta carpeta. Nómbralo así:

```
me1-bitacora.md
me2-bitacora.md
me3-bitacora.md
me4-bitacora.md
me5-bitacora.md
```

## Rúbrica (binaria: cumple / no cumple)

- [ ] Cita un fragmento real del legacy con archivo y línea — no descripción genérica
- [ ] Conecta el patrón a ese fragmento específico, no a una definición del libro
- [ ] Nombra las clases nuevas o modificadas y el rol que cumplen en el patrón
- [ ] Descarta al menos un patrón alternativo con razón válida en este contexto
- [ ] Nombra un trade-off concreto (clases agregadas, complejidad, acoplamiento)
- [ ] Responde la pregunta 5 con algo específico, no con "no cambiaría nada"

5 o 6 criterios cumplidos = aprobada. 0-4 = rechazada, con opción de reentrega en 5 días.
